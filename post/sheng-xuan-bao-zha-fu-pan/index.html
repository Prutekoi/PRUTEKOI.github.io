int C(int n,int m){
	for(int i=0;i<m;i++)d[i]=n-i;
	for(int i=2;i<=m;i++){
		if(vis[i])continue;
		int cnt=0;
		for(int j=i;j<=m;j*=i)cnt+=m/j;
		for(int j=n/i*i;cnt;j-=i){//这里复杂度是mloglogm
			while(d[n-j]%i==0&&cnt)
				d[n-j]/=i,cnt--;//因为cnt总和是m/logm的，所以这里复杂度是m/logm
		}
	}
	int ret=1;
	for(int i=0;i<m;i++)ret=1ll*ret*d[i]%mod;
	return ret;
}

还可以考虑另一种思路：当我们从一个点的所有儿子转移到它自己时，相当于是子树内所有值都 +1+1+1。那么我们考虑 +1+1+1 会造成什么影响。。对于第 iii 位，如果 xmod  2i=2i−1x\mod 2^i=2^i-1xmod2i=2i−1，那么 x+1x+1x+1 会异或上 2i2^i2i。发现了这么一种思路，现在我们关心的只是一个位置的值的个数，那么开桶即可。然后套上dsu即可做到 O(nlog⁡2n)O(n\log^2 n)O(nlog2n)。线段树合并也可以，虽然去掉了dsu的 log⁡\loglog，但是从桶强行套上个线段树的话仍然会多个 log⁡\loglog，仍然是 O(nlog⁡2n)O(n\log^2n)O(nlog2n) 的做法。z7z，skyh都是写的这个做法，但是z7z空间开小了，只拿了 808080。

上面的这种思路还可以优化，即我们发现它的贡献是可减的，那么套上天天爱跑步的trick即可做到 O(nlog⁡n)O(n\log n)O(nlogn)。还有一种树上差分的做法，优化优化之后也变成了这个做法，只不过解释不太一样...cold_chair写的是“树上差分”的做法...然后我们也可以发现我的那个垃圾做法还可以抢救一下，因为它也是可减的，可以做到 O(nlog⁡2n)O(n\log^2n)O(nlog2n)，空间复杂度也可以接受了。

还有一种trie树的做法：考虑一个数 +1+1+1 时是把末尾连续的 111 都变成 000，再把最后面的 000 变成 111。那么我们把数建出01trie，不过它是从低位往高位插的。这样我们就可以 O(nlog⁡n)O(n\log n)O(nlogn) 实现整体 +1+1+1 操作：交换相应的左儿子和右儿子即可。这样我们再写一个trie树合并即可 O(nlog⁡n)O(n\log n)O(nlogn)。这是一道Ynoi的trick。硬核数据结构大王zbh写了这个做法。
